# 项目审计报告（2026-03-10）

## 1. 审计结论

当前仓库已经从“功能堆叠型脚本集合”进化为一套边界相对清晰的本地量化训练系统，主链完整度较高。

### 1.1 当前最健康的部分

- 入口结构清晰：`app/` 统一承接正式实现
- 数据层统一：`market_data/` 已形成 canonical SQLite 主链
- 训练闭环完整：模型、会议、模拟交易、评估、复盘、优化、固化全部打通
- 运行态可审计：周期结果、会议记录、配置快照、training lab、memory 都能落盘
- 多模型实验能力可用：leaderboard 与 allocator 已形成闭环

### 1.2 当前主要风险点

- `SelfLearningController` 仍偏重，流程编排与状态落盘集中在单类中
- Web API 的能力广度已经明显超过当前静态 UI 展示广度
- 仍然是单机/单实例优先设计，未来多实例扩展需要重新设计锁与共享状态
- 仍缺正式 schema migration 机制
- 历史辅助脚本与主链入口虽已区分，但仍需持续约束不要回流

## 2. 当前架构判断

### 2.1 结构已经够稳定，可以继续演化

当前不是“必须推倒重写”的阶段，而是“可以在现有稳定骨架上继续迭代”的阶段。

### 2.2 文档滞后曾经是主要问题

本次文档更新前，文档中仍存在：

- 已不存在的目录名
- 已被替换的旧流程名
- 与当前 `app/` 收口结构不一致的描述

本轮更新后，这个问题已经基本被收口。

## 3. 关键观察

### 3.1 Commander 已经不是包装壳，而是运行时核心

`CommanderRuntime` 现在真正承担：

- 状态统一
- tool calling
- training lab 工件管理
- runtime 锁与任务状态
- memory 与策略基因协同

### 3.2 训练系统已具备实验平台特征

系统不再只是“跑一轮策略回测”，而是已经具备：

- 计划
- 执行
- 评估
- promotion 判断
- baseline 比较
- 结果工件化

### 3.3 数据层统一是本仓库最重要的稳定器

当前训练、Web、状态检查都依赖同一套 canonical schema，这极大降低了系统继续演化时的认知成本。

## 4. 建议优先级

### P1

- 继续拆轻 `SelfLearningController`
- 为 training lab 增加更完整的前端展示
- 为关键数据表/字段演进建立更明确的 migration 约定

### P2

- 增强运行期日志检索与异常定位体验
- 增强 baseline / promotion 协议说明与前端输入校验
- 评估是否将 `sync_data.py` 彻底归档

### P3

- 如果未来要做长时间无人值守运行，再评估更强的多进程/多实例方案

## 5. 最终判断

当前项目已经具备继续做“策略实验平台化”的基础，不需要大拆大改；最值得做的是：

- 在现有主链上继续减重、补观测、补文档、补协议
- 保持数据层和运行时的一致性约束
