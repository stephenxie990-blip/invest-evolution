# 项目问题清单与优先级路线图（2026-03-15）

日期：2026-03-15  
目标：把当前项目从“治理基线已成立的强研究平台”，推进到“结果更可信、过程更可解释、开源更友好的 Agent-first 协作底座”。

---

## 1. 当前阶段判断

当前项目已经具备以下基础：

- 统一运行时骨架成立
- 训练 / 研究 / 运行一体化成立
- Agent 协作主链成立
- promotion / routing / deployment stage / freeze gate 已进入代码与工件

但它还没有达到下面这个状态：

- 结果已经稳定跨过生产级质量门
- 路由、晋级、冻结在所有路径上完全一致
- 开发 / 验证 / 开源入口已经足够轻、足够清楚

所以，下一阶段不应该继续扩故事、扩 buzzword、扩角色数量，而应该集中打几场真正决定系统质量的仗。

---

## 2. 项目问题清单

## P0：必须先解决的问题

### P0-1 治理工件链还没有完全收口

**问题**

- 运行工件之间仍可能存在一致性缺口。
- 部分 resolved policy / gate 结果还没有完整、稳定地回写到最终工件。
- run-specific 工件与 aggregate 工件仍存在“最后一步刷新不彻底”的风险。

**为什么重要**

这是整个系统可信度的地基。  
如果工件之间不能稳定回答“这一轮到底发生了什么、为什么发生、当前处于什么部署阶段”，后续所有优化都会变得难以解释。

**要做什么**

1. 收紧 `cycle result -> leaderboard -> training evaluation -> final summary` 的刷新顺序
2. 把 resolved `promotion_gate / freeze_gate / quality_gate_matrix` 显式写入最终工件
3. 增加强契约测试，确保关键工件字段不漂移

**完成标志**

- 任意一轮训练都能从工件中直接回答：
  - 协议是什么
  - 结果是什么
  - 候选是否生成
  - 当前是 `active / candidate / override`
  - 为什么没进 routing / 没过 promotion / 没过 freeze

### P0-2 效果优化还没有进入治理门内的稳定提升

**问题**

- benchmark pass rate 仍明显偏低
- Sharpe / 回撤 / regime 适配质量尚不稳定
- 最新治理复跑说明：治理变强了，但效果还没有跟上

**为什么重要**

现在系统已经会更谨慎地拒绝不合格候选，但还不会稳定地产出“可以放权”的候选。  
这会让系统长期停留在“会研究、会否决、但不会稳定晋级”的阶段。

**要做什么**

1. 把效果优化目标明确切成：
   - `benchmark_pass_rate`
   - `avg_sharpe_ratio`
   - `avg_max_drawdown`
   - `candidate_ab`
2. 对 `bull / bear / oscillation` 分别做 regime 级目标拆分
3. 优先优化当前主链最有希望的模型组合，而不是同时撒网全部模型

**完成标志**

- 至少一个主模型能在治理门内持续通过 routing eligibility
- 至少一条 candidate 晋级链不再以 `candidate_pruned` 结束
- freeze 失败原因从“系统性失败”收敛到“少数特定指标失败”

### P0-3 路由、晋级、冻结标准还没有完全一体化

**问题**

- review、optimization、routing、promotion、freeze 虽然已经开始共用治理语义，但标准仍没有完全一体化
- 仍有可能出现“上游已经判断不好，下游又从另一个口子把它拉回候选”的情况

**为什么重要**

如果系统内部标准不一致，就会出现“解释是一套、准入是一套、冻结又是一套”的情况。  
这会直接损害 Agent 系统最宝贵的东西：可控性。

**要做什么**

1. 明确统一的 `quality_gate_matrix` 主来源
2. 把 routing / promotion / freeze 的关键门槛做成共享配置与共享解释字段
3. 增加端到端测试，证明“不合格对象不会从别的路径重新变成合格”

**完成标志**

- 系统对一个模型的判断在 review、leaderboard、allocator、routing、promotion、freeze 中保持一致

## P1：下一阶段必须推进，但可以排在 P0 后面

### P1-1 开源边界与 GitHub 入口还需要继续打磨

**问题**

- GitHub 首页叙事还需要持续收口
- 社区文档、贡献说明、安全边界需要更稳定的对外表达
- 新人很容易看见“功能很多”，却不知道“当前真相是什么、这个项目不是什么”

**为什么重要**

开源不是把代码扔上去，而是让外部人能正确理解、正确使用、正确参与。  
如果对外叙事不准，外部反馈会把项目拉向错误方向。

**要做什么**

1. 持续收口 README、Docs Index、Roadmap、Contributing、Security
2. 明确“当前是研究平台，不是实盘托管引擎”
3. 让 GitHub 首页直接回答：
   - 这是什么
   - 为什么值得看
   - 现在处于什么阶段
   - 外部人能怎么参与

**完成标志**

- 第一次打开仓库的人，5 分钟内能看明白项目定位、当前能力、当前阶段、非目标和参与方式

### P1-2 工程边界还不够轻

**问题**

- `invest/__init__.py` 初始化偏重
- 环境依赖对测试 collection 仍较敏感
- 部分跨层对象仍依赖 `dict[str, Any]`

**为什么重要**

这会拉高二次开发、开源协作、测试稳定性和后续重构成本。

**要做什么**

1. 轻量化包初始化
2. 为关键对象逐步补 typed contract
3. 收口标准验证入口，让 `.venv`、CI、本地开发体验一致

**完成标志**

- 开发者可以更轻量地导入和测试关键模块
- 关键协议对象不再大量依赖松散字典

### P1-3 Agent 证据层还不够强

**问题**

- 当前很多判断仍主要体现在 reasoning 文本里
- 还不够容易回答“某个 Agent 为什么给出这个结论、用了哪些事实、对最后结果贡献了什么”

**为什么重要**

如果没有更强的结构化证据层，Agent 系统会长期停留在“看起来合理，但难以严格比较”的状态。

**要做什么**

1. 增加更结构化的 evidence schema
2. 为会议输出补充事实引用和 attribution
3. 提升 Training Lab 对证据链的可视化能力

**完成标志**

- 能够对同一轮协作回答“用了哪些事实、哪个角色起了关键作用、最终为什么被批准/拒绝”

## P2：暂时不要抢先投入的方向

### P2-1 不要优先扩更多 Agent 角色

- 现在的问题不是角色数量不够，而是现有角色的治理、证据与效果还没完全收口。

### P2-2 不要优先做跨场景迁移

- 供应链、物流、其他决策系统都可以迁移，但现在最重要的是先把投资样板场景做扎实。

### P2-3 不要优先引入更重的新模型叙事

- 比如更大的 RL 叠层、更复杂的自主循环、更多工具调用权限，这些都不应早于治理和效果收口。

## 3. 下一阶段最该打哪几仗

### 第一仗：治理工件闭环战

**优先级：最高 / 立即开打**

目标：

- 让每一轮训练从协议到工件到结论都完全自洽

核心动作：

1. 收口 leaderboard 最后刷新
2. resolved gate policy 全量写入工件
3. promotion / lineage / freeze / routing 端到端一致性测试补齐

为什么是第一仗：

- 因为这场仗决定后面所有优化是不是可信

### 第二仗：治理门内效果优化战

**优先级：最高 / 紧随其后**

目标：

- 不是把裸收益做高，而是把“能过治理门”的候选做出来

核心动作：

1. 以 benchmark、Sharpe、回撤为主目标优化
2. 按 regime 拆分优化目标
3. 重点关注 candidate 能否真正晋级，而不是只生成更多 candidate

为什么是第二仗：

- 因为现在系统最真实的瓶颈不是“不会训练”，而是“训练完还不够好，放不了权”

### 第三仗：统一质量标准战

**优先级：高**

目标：

- 让 review、routing、promotion、freeze 的判断标准真正同源

核心动作：

1. 统一 `quality_gate_matrix`
2. 明确各门槛的共享字段和解释字段
3. 堵住“不合格对象从旁路重新上桌”的可能

为什么是第三仗：

- 因为 Agent 系统一旦标准不一致，后面会越来越难解释

### 第四仗：GitHub 开源可信度战

**优先级：高 / 可并行**

目标：

- 让 GitHub 仓库本身成为可信入口，而不是混乱入口

核心动作：

1. README、Docs、Contributing、Security 持续收口
2. 清理旧文档与弱引导
3. 把当前阶段、非目标、贡献方式说清楚

为什么是第四仗：

- 因为你的影响力建设，不只靠代码，也靠别人能不能正确理解这套系统

### 第五仗：工程边界减重战

**优先级：中高**

目标：

- 让仓库更容易被协作、测试、复用和二次开发

核心动作：

1. 轻量化包初始化
2. 增强 typed contracts
3. 统一验证命令和环境约定

为什么是第五仗：

- 因为这决定它能不能真正成为一个被别人接得住的开源项目

## 4. 建议节奏（30 / 60 / 90 天）

## 未来 30 天

只做三件事：

1. 治理工件闭环收口
2. 路由 / promotion / freeze 质量标准统一
3. GitHub 对外文档与开源边界收口

**阶段目标**：从“能讲清方向”升级到“能讲清每次训练到底发生了什么”。

## 未来 60 天

重点进入效果优化：

1. benchmark / Sharpe / 回撤目标优化
2. regime 分层优化
3. candidate 晋级质量优化

**阶段目标**：从“会否决坏候选”升级到“能稳定产出至少一条可信候选链”。

## 未来 90 天

重点进入平台化与开源协作阶段：

1. 工程边界减重
2. 关键协议类型化
3. Agent 证据层强化
4. 更清楚的外部协作入口

**阶段目标**：从“你自己能持续推进的系统”升级到“外部人也能看懂、参与、复用的系统”。

---

## 5. 一句话作战建议

> **下一阶段不要急着把它做得更像“自主交易员”，而要先把它做成一个更可信的、会留下证据、会统一标准、会在治理门内成长的 Agent 系统。**
